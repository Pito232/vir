Benchmark propustnosti sítě v kontejnerech

Komplexní nasazení LXC a Dockeru s dokumentací výsledků z terminálu

Architektura a topologie sítě

Cílem projektu bylo kvantifikovat režii síťové virtualizace a protokolů (NFSv4 a FTP) při distribuci dat mezi kontejnery běžícími na stejném hostiteli (VM Ubuntu 22.04 LTS).

  • Backend (Docker): NFS Server (Alpine Linux) poskytující 800MB testovací soubor. IP 172.17.0.2 (docker0 bridge).
  • Frontend (LXC): FTP Gateway (vsFTPd) čtoucí data z NFS. IP 10.99.99.98 (lxdbr0 bridge).

1 Instalace a konfigurace backendu (Docker)

Příprava dat

Na hostitelském stroji vytvoříme sdílenou složku a vygenerujeme testovací soubor o velikosti 800 MB pomocí náhodných dat, abychom zamezili zkreslení výsledků kompresí na síťové vrstvě.

sudo mkdir -p /srv/nfs_share
dd if=/dev/urandom of=/srv/nfs_share/test_800MB.bin bs=1M count=800

Spuštění NFS serveru

Spustíme Docker kontejner v privilegovaném režimu, který nasdílí připravenou složku přes protokol NFSv4 na výchozím portu 2049.

sudo docker run -d \
  --name nfs-server \
  --privileged \
  -v /srv/nfs_share:/nfsshare \
  -e SHARED_DIRECTORY=/nfsshare \
  -p 2049:2049 \
  itsthenetwork/nfs-server-alpine:latest

2 Instalace a konfigurace frontendu (LXC)

Inicializace LXD a spuštění kontejneru

Při inicializaci LXD nastavíme velikost storage poolu rovnou na 5 GB, abychom zajistili dostatek místa pro systém a testovací soubory.

sudo lxd init
lxc launch ubuntu:22.04 ftp-kontejner

Povolení privilegovaného režimu pro NFS mount:

lxc stop ftp-kontejner
lxc config set ftp-kontejner security.privileged true
lxc config set ftp-kontejner raw.lxc "lxc.apparmor.profile=unconfined"
lxc start ftp-kontejner

Instalace služeb a připojení disku

Vstoupíme do kontejneru, nainstalujeme potřebné služby a namountujeme NFS sdílení z IP adresy Docker bridge (172.17.0.1).


lxc exec ftp-kontejner -- bash
apt update && apt install nfs-common vsftpd -y
mkdir -p /mnt/nfs_remote
mount -t nfs 172.17.0.1:/ /mnt/nfs_remote

Konfigurace FTP serveru (vsftpd)

Pro účely testování zprovozníme anonymní přístup. Pro zabezpečení (chroot jail) vsftpd vyžaduje, aby anonymní kořenová složka neměla práva k zápisu a neprobíhala kontrola PAM.

# Otevřeme /etc/vsftpd.conf a nastavíme:
listen=YES
listen_ipv6=NO
anonymous_enable=YES
no_anon_password=YES
anon_root=/mnt/nfs_remote
check_shell=NO

# Deaktivace PAM autentizace
sed -i 's/^pam_service_name=vsftpd/#pam_service_name=vsftpd/' /etc/vsftpd.conf

# Zabezpečení složky a restart
chown root:root /mnt/nfs_remote
chmod 555 /mnt/nfs_remote
systemctl restart vsftpd
Konfigurace vsFTPd: Pro FTP přístup k NFS datům byl v /etc/vsftpd.conf nastaven anon_root=/mnt/nfs_remote a složka byla zabezpečena proti zápisu (chmod 555). PAM autentizace byla deaktivována.

3 Měření a Dokumentace z Terminálu

Testování bylo rozděleno na tři scénáře, aby byl jasně vidět úbytek výkonu způsobený jednotlivými vrstvami.

Scénář testu Cesta datového toku Naměřená rychlost (Průměr)
A. Čistá síť (iPerf3) Host RAM → Bridge → RAM LXC 44.3 Gbits/s
B. Lokální FTP přenos LXC Disk → vsFTPd → Bridge → Hostitel 100.0 MB/s
C. Distribuovaný přenos (NFS) Docker → NFS → Bridge → LXC → FTP → Hostitel 88.9 MB/s

A. Ukázka čisté síťové propustnosti (iPerf3)

Toto měření dokazuje, že virtuální mosty mezi kontejnery nejsou úzkým hrdlem systému.

4 Závěr

NFS Overhead: Srovnáním testu B (lokální disk, 100 MB/s) a testu C (NFS disk, 88.9 MB/s) jasně identifikujeme úbytek výkonu způsobený NFS vrstvou. Pokles představuje cca 11,1 % reálné propustnosti při započtení diskových operací.

Závěr: Projekt úspěšně demonstroval, že kontejnerizace služeb (LXC a Docker) na stejném hostiteli je vysoce efektivní, pokud jsou virtuální sítě správně routovány. Distribuce dat přes NFS sice přináší měřitelný pokles propustnosti kvůli enkapsulaci paketů a přepínání kontextu mezi kontejnery, ale naměřených 89 MB/s je vynikající kompromis pro škálovatelnou mikroslužbovou architekturu.
